# terrace-k-dashboard SPEC

## 概要
Terrace.Kチームの稼働状況をブラウザで一覧できる管理画面に、**チーム共通のタスク状態一覧**を追加し、誰がボールを持っているか・何が止まりかけているかを把握しやすくする改善を行う。

## 目的
- 各エージェントの現在の稼働状況をひと目で把握できるようにする
- current-task とは別軸で、チーム共通のタスク状態を一覧できるようにする
- 「誰がボールを持っているか」「次に何をするか」「更新が止まっていないか」を見失わないようにする
- projects.md と将来的に自然に接続できる構造を意識しつつ、まずは壊れない最小運用から始める

## 今回の改善対象
既存の terrace-k-dashboard に対して、以下を追加・微修正する。

1. **状態定義の固定**
2. **チーム共通の状態一覧ファイルの新設**
3. **状態一覧を読む最小ビューの追加**
4. **裏側の微修正（必要最小限）**

## 状態定義（固定版）
- **未着手**：タスクは存在するが、担当者がまだ実作業を始めていない
- **進行中**：担当者が現在そのタスクの実作業を進めている
- **依頼待ち**：次担当や関係者へ渡すべきアクションがまだ自分側で未実行
- **返答待ち**：必要な依頼・送信は済んでおり、相手や外部からの返答待ち
- **完了**：必要作業と所定の報告まで完了している
- **問題発生**：詰まり・不明点・判断待ちがあり、そのままでは前進できない

### 状態運用の補助ルール
- 送る前は「依頼待ち」
- 送った後は「返答待ち」
- 報告前は「完了」にしない
- 詰まったら「進行中」のまま抱えず「問題発生」に移す

## 断線分類（運用観点）
- **未着手断線**：指示を受けたのに着手していない
- **受け渡し断線**：着手・完了したのに次担当へ渡っていない
- **可視化断線**：進んでいるのに状態更新や一覧反映がなく見えない
- **報告断線**：完了したのに報告されていない
- **判断断線**：問題発生のまま相談先に上がらず止まっている

## チーム共通状態一覧の最小項目
### 必須項目
- タスク名
- 担当者
- 状態
- 次の一手
- 最終更新時刻

### 補助項目
- **待ち先** を優先採用する
  - 理由：今の課題は「なぜ止まっているか」の分析より先に、「今ボールが誰にあるか」を見失わないことだから
- ブロッカーは今週は入れない

## 試作の境界（今週やること）
- 既存 current-task は置き換えない
- チーム共通の状態一覧ファイルを新設する
- その一覧を読む最小ビューをダッシュボードに追加する
- 一覧の更新は**手動更新前提**で開始する
- 保存形式は **JSON** を第一候補とする

## 今週やらないこと
- current-task からの自動反映
- Slackログ解析による状態更新
- 断線の自動判定
- dashboard本体への大規模な統合や全面改修
- 自由記述のブロッカー管理

## 成功条件（今週）
1. 状態一覧ファイルの形式が固定されている
2. 人間が手で更新できる
3. その内容を一覧として見られる
4. current-task と併存しても混乱なく読める

## 想定画面構成（追加後）
- ヘッダー
  - 画面タイトル
  - 最終更新時刻
  - 手動更新ボタン
- エージェント一覧セクション
  - 4人分のカードまたはテーブル
  - 各カードに名前、状態、モデルを表示
- **チーム状態一覧セクション（新設）**
  - 一覧テーブルまたはカード群
  - 表示内容：タスク名 / 担当者 / 状態 / 次の一手 / 待ち先 / 最終更新時刻
  - 見せ方の優先順位：
    1. 止まっているものが浮く
    2. 次の一手が空欄のものが分かる
    3. 更新が古いものが拾える
- Cronジョブ一覧セクション
  - ジョブ名
  - 状態
  - 最終実行時刻
- Gateway状態セクション
  - 稼働中 / 停止 / 異常 など

## データ方針
- 既存の current-task データはそのまま維持する
- 新しいチーム状態一覧は**別ファイル**で管理する
- 初期は手動更新前提でよい
- 後から projects.md や既存ダッシュボードとの自然な接続を検討できる形にする

## 役割分担
- **澪**：状態定義の固定、運用観点の仕様整理、全体進行
- **ナナセ**：最小一覧の情報設計、表示優先度の定義、DESIGNSPEC整理
- **ユイ**：状態一覧JSONのたたき台、最小ビュー案、裏側の微修正含む実装
- **レイン**：全体整合、詰まり・判断断線の整理補助

## デザイン方針
- 情報の網羅性より「止まりかけを拾えること」を優先する
- 管理画面として見やすさ・情報優先・軽量さを重視する
- きよぴさんの好みに合わせ、具体的で把握しやすいUIを優先する
- 既存ダッシュボードのトーンを極力崩さない

## 実装方針（補完仕様）
- 最小構成ではPCブラウザでの閲覧を優先する
- 1画面完結を維持する
- スマホで極端に崩れない配慮は行う
- 状態一覧は既存UIに強く埋め込まず、まずは「横に1枚足す」考え方で実装する
- 状態名や項目名の変更に耐えやすい構造を優先する
