# terrace-k-dashboard DESIGNSPEC

## 1. 目的
Terrace.K Dashboard に、既存の軽量で見やすい管理画面トーンを崩さず、**チーム共通のタスク状態一覧**を追加する。

今回の改善の目的は、単に情報を増やすことではなく、以下を一目で分かるようにすること。

- 誰が今ボールを持っているか
- 何が止まりかけているか
- 次に何をするべきか
- 更新が古いものがどれか

current-task は各エージェントの「いま何をしているか」を示す個人表示として残し、今回追加する状態一覧は**チーム全体の流れを見る別レイヤー**として設計する。

---

## 2. 今回のデザインで達成したいこと
画面を開いたときに、既存の Agents / Cron Jobs / Gateway の流れを保ったまま、新しい状態一覧が自然に読める状態を作る。

達成したいことは以下。

1. エージェント個人の現在作業と、チーム全体のタスク状態が混ざらずに読める
2. 「進んでいるもの」よりも「止まりかけているもの」が先に目に入る
3. 次の一手が空欄のもの、更新が古いものが拾いやすい
4. 新しい情報を足しても、既存ダッシュボードの整理感・軽さを壊さない
5. 実装が複雑になりすぎず、今週の最小試作として成立する

---

## 3. 全体デザイン方針

### 3-1. 基本方針
- **一覧性優先**：個々の説明量より、全体を見渡したときの把握しやすさを優先する
- **止まりかけ優先**：順調なものの網羅より、滞留・空欄・古い更新を拾えることを重視する
- **既存トーン維持**：現在の白基調・淡い境界線・軽いカード感を崩さない
- **役割分離**：current-task と状態一覧の役割を混ぜない
- **実装しやすさ優先**：複雑なフィルター、アニメーション、インライン編集は入れない

### 3-2. 情報の優先順位
今回追加する状態一覧セクションでは、優先順位を以下とする。

1. 状態
2. タスク名
3. 次の一手
4. 担当者
5. 待ち先
6. 最終更新時刻

理由：今回の一覧の目的は「全部を説明すること」ではなく、「止まりかけを拾うこと」だから。特に状態・次の一手・更新時刻が弱いと、一覧の意味が薄くなる。

---

## 4. 画面全体構成
画面の大枠は既存構成を維持する。

1. ヘッダー
2. エージェント一覧
3. **チーム状態一覧（新設）**
4. Cronジョブ一覧
5. Gateway状態

### 4-1. 構成意図
- Agents は「個人の現在地」
- Team Status は「チーム全体のボール位置」
- Cron / Gateway は「仕組みの稼働状態」

この順にすることで、
**人 → タスク → 仕組み**
の読み順になり、自然に状況把握できる。

### 4-2. 状態一覧の位置
新設する「チーム状態一覧」は、**Agents セクションの直後**に置く。

理由：
- まず誰が動いているかを見て
- 次に何のタスクがどう流れているかを見る
- そのあとでジョブやインフラを見る

という順番が運用上もっとも理解しやすいから。

---

## 5. チーム状態一覧セクション仕様

### 5-1. セクション名
見出し候補は以下のどちらか。

- `Team Status`
- `チーム状態一覧`

既存UIが英語見出し寄りなので、第一候補は **Team Status**。
説明文は日本語で補う。

### 5-2. セクション説明文
推奨：
`誰がボールを持っているか、どこが止まりかけているかを確認する一覧`

トーンは既存の section-description に揃え、説明しすぎず役割だけを明確にする。

### 5-3. 表示形式
**初期実装はテーブル形式を推奨**する。

理由：
- 項目が固定されている
- 比較しやすい
- 更新古さや空欄を横並びで見つけやすい
- 実装が軽い

カード形式は見た目の余白は作りやすいが、今回の用途では一覧比較に不利。

---

## 6. 一覧項目仕様

### 6-1. 必須項目
以下をこの順で並べる。

1. タスク名
2. 担当者
3. 状態
4. 次の一手
5. 待ち先
6. 最終更新時刻

※仕様上の核は「タスク名 / 担当者 / 状態 / 次の一手 / 最終更新時刻」だが、今回の運用課題に対しては補助項目としての **待ち先** の効果が高いため、テーブル上は最初から表示してよい。

### 6-2. 列順の理由
UI上の推奨列順は以下。

| 列 | 役割 |
|---|---|
| タスク名 | 何の話かを最初に認識する |
| 担当者 | 誰のタスクかを認識する |
| 状態 | いま何段階かを判断する |
| 次の一手 | 止まり方・次の行動を把握する |
| 待ち先 | 誰にボールがあるかを見る |
| 最終更新時刻 | 更新の古さを判断する |

ただし視覚的な強調は、列順とは別に **状態 / 次の一手 / 更新古さ** にかける。

### 6-3. テキストの長さ制御
- タスク名：1行、省略可
- 担当者：短い固定値、折返し不要
- 状態：バッジ表示
- 次の一手：1〜2行まで許容、長文は省略
- 待ち先：短い固定値想定
- 最終更新時刻：1行固定

### 6-4. 空欄時の扱い
特に重要なのは「次の一手」が空欄のケース。

- 単なる `—` ではなく、**弱い警告表現**で見せる
- 文言候補：`未設定`
- 文字色は warning 系、背景は付けなくてよい

理由：今回の一覧で最も拾いたい異常のひとつが「次の一手未設定」だから。

---

## 7. 状態表示のルール

### 7-1. 状態は必ずバッジ化する
6状態はすべて、テキストではなく既存の status-badge 系UIで表現する。

理由：
- 一目で判別しやすい
- Agents / Cron / Gateway とトーンが揃う
- 色で「順調 / 注意 / 異常」を直感的に把握できる

### 7-2. 状態と色の対応
推奨は以下。

- **未着手**：neutral / gray
- **進行中**：online / green
- **依頼待ち**：warning / amber
- **返答待ち**：warning / amber（依頼待ちより少し弱くてもよいが、初期は同系統で可）
- **完了**：stopped or neutral ではなく、success 系の穏やかな green
- **問題発生**：error / red

### 7-3. 注意点
- 依頼待ち と 返答待ち は意味が異なるが、初期は色差よりラベル差で見分ける
- 問題発生だけは明確に赤で目立たせる
- 完了は目立たせすぎない。終わったものより、止まりかけに視線を渡したい

---

## 8. 止まりかけを拾うための見せ方

### 8-1. 優先して浮かせるべきもの
この一覧で先に視認したいのは以下の3つ。

1. 状態が `問題発生`
2. `次の一手` が未設定
3. `最終更新時刻` が古い

### 8-2. UIでの見せ方
#### 問題発生
- 状態バッジを赤
- 行全体にごく薄い赤背景を敷いてもよい
- ただし強すぎる背景色は全体トーンを壊すので避ける

#### 次の一手未設定
- セル内文言を `未設定`
- 文字色を amber / muted red 系にする
- 背景は不要、太字も不要

#### 更新が古い
- 最終更新時刻の文字色をやや警告寄りにする
- もしくは時刻の横に小さな `更新古め` ラベルを付けてもよい
- 初期実装では、単純に warning トーンで十分

### 8-3. やりすぎないこと
- 行全体を強い色で塗らない
- 複数の警告表現を同時に重ねすぎない
- スコア表示や優先度番号は入れない

目的は監視盤ではなく、軽量な運用一覧であることを守る。

---

## 9. current-task との関係整理

### 9-1. 役割の違い
- current-task：各エージェントが「今なにをしているか」
- Team Status：チーム全体で「どのタスクがどこにあるか」

この違いがUIで読み分けられるようにする。

### 9-2. 読み分けのための見せ方
- Agents セクションは従来どおりカード中心
- Team Status はテーブル中心
- Agents の説明文は「現在の接続状態と利用モデル」
- Team Status の説明文は「ボール位置と停滞把握」

同じ「タスク」という語が出ても、表現形式を変えることで混同を防ぐ。

---

## 10. レイアウト詳細

### 10-1. テーブル幅
PC閲覧優先で、横スクロール可能なテーブルで問題ない。

推奨：
- `table-wrap` を再利用
- `min-width` は 860〜960px 程度

### 10-2. 列幅の目安
- タスク名：やや広め
- 担当者：狭め
- 状態：狭め
- 次の一手：最も広め
- 待ち先：狭め
- 最終更新時刻：中程度

### 10-3. 行高
Cronテーブルと同程度でよいが、次の一手だけ2行を許容する場合は少しだけ余裕を持たせる。

### 10-4. セクション高さ感
この一覧は内容次第で増えるため、固定高さにせず自然高さとする。
ただし、1画面の軽さを維持するため、過度な余白は入れない。

---

## 11. タイポグラフィ

### 11-1. セクション見出し
既存の section-title / section-description を踏襲。

### 11-2. テーブル本文
- タスク名：13〜14px / 600寄り
- 担当者：13px / 500
- 次の一手：13px / 400
- 待ち先：12〜13px / 500
- 最終更新時刻：12〜13px / muted

### 11-3. 空欄・注意表現
- `未設定` は 12〜13px
- 太字にしすぎない
- 色で軽く拾わせる

---

## 12. モバイル・狭幅配慮
今回はPC優先だが、極端に崩れないように以下を守る。

- テーブルは横スクロール可
- 無理にカード化しない
- セクション見出しと説明文は折り返し許容
- 既存のレスポンシブ挙動に合わせる

---

## 13. 実装優先順位

### 必須
1. Team Status セクション新設
2. 状態一覧テーブル追加
3. 状態バッジの表示ルール整備
4. 次の一手未設定の軽い警告表示
5. 更新古さを拾える最低限の表現

### できれば合わせて整える
1. 問題発生行のごく薄い背景強調
2. 待ち先の表示トーン最適化
3. セクション説明文の微調整

### 今回やらない
1. テーブル内編集
2. 並び替え・フィルター
3. 自動色判定の複雑化
4. current-task との自動連携表現
5. ブロッカー自由記述UI

---

## 14. ユイ向け実装要約

### 追加するセクション
- Agents の次、Cron Jobs の前に `Team Status` セクションを追加

### 表示形式
- テーブル形式
- 列は `タスク名 / 担当者 / 状態 / 次の一手 / 待ち先 / 最終更新時刻`

### 見せ方
- 状態はバッジ
- `問題発生` は最も目立つ
- `次の一手` が空欄なら `未設定` 表示で弱警告
- 更新が古いものは軽く warning 表現
- 全体トーンは既存の Cron テーブルと揃える

### 重要な方針
- current-task は残す
- 新しい一覧は別レイヤーとして読ませる
- 「全部を説明するUI」ではなく「止まりかけを拾うUI」にする

---

## 15. このデザイン判断の意図
今回の改善で最も避けたいのは、情報を追加した結果、
**“何が大事なのか逆に分からなくなること”**。

そのため、状態一覧は多機能な管理表ではなく、
**チームの流れが詰まりそうな場所を先に見つけるための薄いレイヤー**
として設計する。

- 状態で段階を把握する
- 次の一手で前進可能性を確認する
- 待ち先でボール位置を見る
- 最終更新時刻で古さを拾う

この4点が自然に読めれば、今週の試作としては十分価値がある。

既存ダッシュボードの軽さと整理感を守りながら、運用の見通しを一段上げることが今回の設計意図である。
